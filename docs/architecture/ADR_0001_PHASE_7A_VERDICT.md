## Fixture description

Two datasets of 10 000 entries each, generated by `benchmarks/bench_phase_7a_action_index.py` with fixed seeds, materialised as monolithic JSONL (`shards/sessions.jsonl`) under disposable `tempfile.mkdtemp(...)` roots — same layout as `tests/rr/conftest.py:56–71`. Parameters, anchor points:

- `N_ENTRIES = 10_000` (`benchmarks/bench_phase_7a_action_index.py:54`)
- `N_SESSIONS = 500` (line 55) → ~20 entries per session on average.
- `TIME_SPAN_DAYS = 30` (line 56), timestamps drawn uniformly over the window starting `2026-03-01T00:00:00+00:00` (line 110).
- Event-type mix (line 57–63): 70 % `tool_call`, 15 % `note`, 5 % each of `session_start`, `snapshot`, `session_end`. All `tool_call` entries carry an `action_name`; non-`tool_call` entries carry one with 2 % probability to mirror real-traffic noise (line 141).
- Dataset A (`benchmarks/bench_phase_7a_action_index.py:66–72`) — **30 action_names**, Zipf-like weights `w(rank) = 1 / (rank+1)^1.1` (line 79), seed `42`. 30 distinct actions observed.
- Dataset B (`benchmarks/bench_phase_7a_action_index.py:75–81`) — **1 000 action_names**, uniform weights, seed `43`. 997 distinct actions observed in 10 000 entries.

Deterministic pick of query targets per dataset (`benchmarks/bench_phase_7a_action_index.py:145–161`):
- `top_action` = rank-1 action by observed count.
- `rare_action` = action at the 90th-percentile rank (bottom 10 %).
- `c1_session` = the session with the most `top_action` occurrences (ties broken alphabetically).
- `c2_start` / `c2_end` = `t0 + 10 days` / `t0 + 20 days` (middle third of the time span, line 163–164).

Measurement protocol (`benchmarks/bench_phase_7a_action_index.py:187–210`):
- Build series: **5 runs per metric**, each on a fresh `tmp_path` re-materialised from the same generator — stats published as **median** (gate basis) with p95 and max for transparency.
- Query series: 5 warmup runs followed by **100 timed runs** (line 210), indexes built once before the series and reused — stats as median / p95 / max.

## Verdict

**Final verdict: PASS — on BOTH datasets, on BOTH gates.**

Both PASS conditions of the Phase 7a exit criterion are met:
- `delta_build ≤ 1× SessionIndex_build` on dataset A (**0.79×**) and dataset B (**0.78×**). The incremental cost of promoting `action_name` into an RR index is well below the budget. `delta_build` was computed from medians of 5-run series (not from single-shot measurements), so the signal is not bruit-dominated.
- All four query variants (top / rare / C1 / C2) are `≤ 1.5× SessionIndex` on both datasets. In fact **every** variant on **every** dataset is **faster than SessionIndex** — the worst RR ratio on any variant is 0.69× (dataset A, top query), far below the gate.

The RR dict-bucket structure wins structurally on rare queries (early exit is useless for SessionIndex when the match is sparse in a long flat list), on session-combined queries (the bucket is already small before the AND-filter), and on time-combined queries (the bucket is already sorted by timestamp).

### Operational concerns (do not block PASS)

**RR_with_action_build absolute ratio = 6.44× (dataset A) / 6.41× (dataset B) — above the 3× informational threshold → flagged.**

This is *not* a new cost introduced by Phase 7a. RR_baseline (without the action index) is already 5.63–5.65× SessionIndex (`benchmarks/results/phase_7a_action_index_20260419.json:datasets[*].builds.RR_baseline_build.median_ms`). The flag reflects a structural trade-off that ADR 0001 already accepted under driver 8 (Operational cost): RR maintains four cross-cutting indexes (session / agent / timeline / shard), SessionIndex maintains one specialised one. The Phase 7a extension only lifts the ratio from ~5.65× to ~6.43× — a ~14 % relative increase on an already-large baseline, not a step change.

**Mitigation note for migration.** Phase 7b operators should schedule index rebuilds on meaningful shard growth (as they already do for SessionIndex today, `src/dsm/cli.py:570–580 dsm session-index`), not at read time. In the long run RR benefits from an incremental build (not in scope for this prototype) — covered by a dedicated follow-up ADR if/when absolute build time becomes a production bottleneck. Until then, 550 ms to rebuild over 10 000 entries is a one-time operator cost, not a per-query cost.

### Consequences of this verdict

- **Phase 7b is unblocked.** Rebranching the 8 live SessionIndex consumers (listed in `docs/architecture/ADR_0001_SESSIONINDEX_CLASSIFICATION.md` under the "Implications" section) onto `RRQueryEngine.query_actions` is mechanically feasible within the ADR 0001 exit-criterion budget.
- **ADR 0001 remains `Proposed`.** A PASS verdict satisfies a necessary precondition for promotion; it does not itself promote. The transition `Proposed → Accepted` is a separate phase that weighs the full migration plan, not just Phase 7a.
- **`SessionIndex` classification does not need to be re-executed.** The `duplicative` verdict from `docs/architecture/ADR_0001_SESSIONINDEX_CLASSIFICATION.md` stands: RR is empirically demonstrated as able to cover the one access pattern (`action_name` filter) that was observed missing at criterion (a) review, at equal-or-better latency on all query shapes tested.

### Commitments by this verdict

- The prototype branch `proto/phase-7a-rr-action-name-index` is **not merged** into `main`. It carries exactly three code commits on top of `main` (`docs` + `proto` + `bench`) and is to be treated as Phase 7a's replayable artefact.
- No production consumer of SessionIndex has been touched. Phase 7b owns that work.
- No document outside `docs/architecture/` and no global doc (`README.md`, `ARCHITECTURE.md`, etc.) was modified, per the Phase 7a scope.
