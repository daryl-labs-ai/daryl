# ADR 0001 — Storage.read Scaling Probe

- **Date:** 2026-04-20
- **Parent ADR:** `docs/architecture/ADR_0001_CANONICAL_CONSUMPTION_PATH.md`
- **Related:** `docs/architecture/ADR_0001_PHASE_7A_5_ROOTCAUSE.md` (identified `storage_read_batch` as 70 % of RR build cost and listed internal behaviour of `Storage.read` as the rank-1 unmeasured zone).
- **Status of parent ADR:** Proposed (unchanged).
- **Kernel code:** untouched (freeze `2026-03-14` respected — `git diff 58d7789..HEAD -- src/dsm/core/` returns empty).
- **Purpose:** quantify `Storage.read(shard_id, offset=K, limit=N)` scaling under varying `K`, in isolation from RR, to determine whether the 70 % `storage_read_batch` contribution is fixable within RR (N+1A scope) or requires kernel reopening (a separate ADR, out of scope here).
- **No code-level optimization is proposed in this report.** Section "Strategic implication" branches conditionally on the measured verdict as authorised by the probe brief.
- **Run artefacts:** `benchmarks/results/storage_read_probe_20260420.json` + `.md`. Generated by `benchmarks/bench_storage_read_scaling.py` on branch `proto/phase-7a-rr-action-name-index` @ `9967c87`.

## Probe design (recap)

- Fixture size : **100 000 entries**, seed **44**, Entry-schema-compatible JSON.
- Two shard layouts tested independently on fresh `tempfile.mkdtemp()` dirs :
  - **Segmented** (dispatches to `src/dsm/core/storage.py:163` `_read_segmented`) — 10 segments of 10 000 entries each (matches kernel `MAX_EVENTS_PER_SEGMENT = 10_000` at `src/dsm/core/shard_segments.py:33`).
  - **Monolithic** (dispatches to `src/dsm/core/storage.py:193` `_read_monolithic`) — single `shards/<id>.jsonl` with all 100 000 entries.
- Main sweep : offsets `[0, 1 000, 10 000, 50 000, 90 000, 99 900]` at fixed `limit=100`.
- Secondary sweep (limit sensitivity at fixed `offset=50 000`) : limits `[10, 100, 1 000]`.
- 5 warmup + 50 timed runs per point. `time.monotonic()`.
- Fingerprint stability verified : for every `(mode, offset, limit)` the `sha256` of returned entry ids was identical across all 50 runs. **All points fingerprint-stable** — no non-deterministic behaviour observed.
- Classifier (from brief) :
  - all Δ/K within ±20 % of mean → `flat` or `linear-like` depending on magnitude ;
  - Δ/K monotonically increasing → `super-linear-like` ;
  - otherwise → `other / unclear`.

### Slope analysis (segmented)

Baseline at K=0 : **2.7346 ms** (one segment read, 10 000 lines parsed).

| Offset K | Δ vs K=0 (ms) | Δ / K (µs per offset skipped) |
|---:|---:|---:|
| 1 000 | 2.26 | 2.26 |
| 10 000 | 25.68 | 2.57 |
| 50 000 | 129.02 | 2.58 |
| 90 000 | 231.28 | 2.57 |
| 99 900 | 254.62 | 2.55 |

Mean of the four "converged" points (K ≥ 10 000) : **2.567 µs / offset**, spread 0.03 µs (1.2 % of mean). The K=1 000 point sits 12 % below the mean — still within the ±20 % rule.

**Classifier verdict — segmented : `linear-like`.**

### Interpretation (segmented)

The cost per entry skipped is empirically constant at **~2.55 µs / offset** across the tested range, plus a fixed baseline of ~2.7 ms for reading the newest segment. The constant-slope behaviour is consistent with the code structure observed at `src/dsm/core/storage.py:170–191` — `_read_segmented` iterates newest-segment-first, loads each segment fully into memory via `f.readlines()`, then iterates `reversed(lines)` with a `skipped < offset` counter that increments per entry until offset is reached. The per-entry work in the skip path is one `json.loads` + one counter increment ; 2.55 µs matches the expected cost of that loop body at CPython speed.

At K=99 900, all 10 segments are opened and fully parsed (we return the oldest 100 of the shard) → total 100 000 lines parsed × ~2.5 µs per parse = ~250 ms, matches the observed 257 ms within 3 %.

### Limit sensitivity (segmented, fixed offset=50 000)

| Limit | median (ms) |
|---:|---:|
| 10 | 130.08 |
| 100 | 131.90 |
| 1 000 | 135.62 |

Cost is nearly invariant to limit at fixed offset : going from `limit=10` to `limit=1 000` adds only ~5.5 ms. The per-call cost is **dominated by the skip phase, not by the return phase**. This confirms that the linear-like slope identified above is driven by offset-skip work, not by the size of the returned batch.

### Slope analysis (monolithic)

Baseline at K=0 : **353.8337 ms**.

| Offset K | Δ vs K=0 (ms) | Δ / K (µs per offset skipped) |
|---:|---:|---:|
| 1 000 | 2.76 | 2.76 |
| 10 000 | 6.79 | 0.68 |
| 50 000 | 136.35 | 2.73 |
| 90 000 | 255.14 | 2.83 |
| 99 900 | 175.25 | 1.75 |

Δ/K ratios : `[2.76, 0.68, 2.73, 2.83, 1.75]`. Mean ≈ 2.15 µs. Spread 2.15 µs = 100 % of mean (far outside ±20 %). Not monotone-increasing either (2.76 → 0.68 is a drop ; 2.83 → 1.75 is a drop). Classifier rule therefore triggers `other / unclear`.

**Classifier verdict — monolithic : `other / unclear`.**

### Interpretation (monolithic) — qualified reading

The mechanical label `other / unclear` is the correct output of the brief's rule but it understates the actual signal. The empirical pattern is :

- The **K=0 baseline is 354 ms** — i.e. even with zero entries skipped, a call to `Storage.read` on a 100 000-entry monolithic shard costs ~354 ms. That cost is the full-file `json.loads` scan inherent to `_read_monolithic` at `src/dsm/core/storage.py:198–225`, which iterates `for line in f:` over the entire file irrespective of the `offset` parameter, keeping entries in a `deque(maxlen=offset + limit)`.
- The deltas at higher offsets (K=50k → +136 ms, K=90k → +255 ms) reflect the growing `deque` size (it holds more entries as offset grows) plus the final `list(window)[start:end][::-1]` slice/reverse cost.
- The K=99 900 regression (529 ms, lower than K=90 000's 609 ms) is measurement jitter, not a real offset-driven effect — consistent with the high run-to-run variance shown at K=10 000 (stdev 157 ms, p95/max fanning out to 640 / 1150 ms) from GC or FS-cache timing.

So the honest physical reading is : **per-call cost on monolithic is dominated by a fixed O(file_size) term (~354 ms at 100 k entries), with a weaker offset-driven component**. The mechanical classifier's `other / unclear` output reflects the fact that the mean-reversion noise on the Δ/K ratios is comparable to any offset-driven trend. This probe does not cleanly isolate the smaller offset-scaling term against the much larger file-scan baseline.

## Verdict

- **Scaling verdict — segmented : `offset-sensitive (empirically linear-like)`.**
  Quantification : **time per entry skipped ≈ 2.55 µs / offset**, stable across K ∈ [10 000, 99 900] with < 3 % spread of Δ/K.

- **Scaling verdict — monolithic : `other / unclear`.**
  Under the mechanical classifier — the Δ/K ratios oscillate (0.68–2.83 µs) without a constant or monotone signature. The physical reading, not produced by the classifier but supported by the limit-sensitivity sweep and by reading `_read_monolithic` at `src/dsm/core/storage.py:198–225`, is that per-call cost is **dominated by a fixed O(file_size) term ~354 ms at 100 k entries**, with a smaller offset-driven component buried in timing noise. `Storage.read` on monolithic is **not offset-sensitive in any meaningful sense — the offset-scaling signal is overwhelmed by the per-call full-file-scan constant**.

> **Methodological note.** This probe classifies empirical scaling behaviour under the tested offsets (0, 1 k, 10 k, 50 k, 90 k, 99.9 k) on a single fixture size (100 k entries). It does not prove asymptotic complexity in the formal sense. A formal asymptotic claim would require multiple fixture sizes and a regression analysis across them ; that is out of scope for this probe. The labels above describe *observed patterns*, not proven complexity classes.

## Strategic implication for ADR 0001

The Phase 7a.5 fixture creates **monolithic** shards (via `benchmarks/bench_phase_7a_action_index.py:113` `_materialize_dataset`, which writes a single `shards/sessions.jsonl` — no segmentation). The 70 % `storage_read_batch` contribution measured in `ADR_0001_PHASE_7A_5_ROOTCAUSE.md` was therefore produced by RR calling `_read_monolithic` repeatedly.

Direct arithmetic cross-check against the probe :

- Phase 7a.5 100 k build makes **21 batch reads** per build run (per `ADR_0001_PHASE_7A_5_ROOTCAUSE.md §A.1`, `samples_per_run_median=21` for `build:storage_read_batch`).
- Probe monolithic per-call cost at offset near zero : ~350–360 ms.
- Expected cumulative cost : 21 × ~350–500 ms ≈ 7 350 – 10 500 ms.
- Observed `storage_read_batch` total on Phase 7a.5 Dataset A : **10 343 ms** (within the computed envelope).
- SessionIndex makes **1 call** with `limit=10**7` (`src/dsm/session/session_index.py:51`) → pays the ~354 ms file scan once.
- Ratio of RR calls to SessionIndex calls : 21 × ≈ 7.4 s of RR overhead that SessionIndex does not pay.

The 70 % contribution is arithmetically explained by RR's paginated call pattern on a monolithic shard, not by a per-call kernel anomaly. Each call's cost is what the kernel's design on monolithic shards specifies : `O(file_size)` per call, not `O(limit)`. RR pays that cost 21 times ; SessionIndex pays it once.

### Strategic branch mapping

The probe brief defined three outcome branches. The measurement straddles two of them depending on the shard layout :

**Branch — segmented ⇒ `offset-sensitive (empirically linear-like)` (from the brief) :**
On segmented shards, per-call cost grows with offset at ~2.55 µs/entry. RR's paginated pattern with increasing offset would accumulate linearly : batch K at offset K×5 000 costs ~K × 5 000 × 2.55 µs ≈ K × 12.75 ms. Sum over 20 batches ≈ 12.75 × (0+1+…+19) × 1 ms ≈ 2 423 ms. A segmented-shard RR build at 100 k would therefore pay ~2.5 s in `storage_read_batch` alone, substantially less than the ~10 s observed on monolithic but still non-trivial.

Per the brief's branch for this case :
> the 70 % contribution is **unlikely to be eliminated** within RR at 100 k scale without one of two interventions : (a) changing RR's call pattern to `Storage.read` materially (fewer calls, larger batches, or a fundamentally different traversal), or (b) changing kernel read behavior to make offset-skip cheaper.

The probe does not prescribe (a) vs (b). For segmented, both options remain open.

**Branch — monolithic ⇒ `other / unclear` (per the brief's fallback category) :**
On monolithic shards, the dominant term is a fixed O(file_size) per-call cost, not an offset-scaling term. The aggregate RR cost scales with **number of calls × file_size**. RR makes 21 calls where SessionIndex makes 1 ; this is a call-pattern difference, not a kernel-scaling difference.

Per the brief's branch for this case, mapping the "flat / constant per call, with call count being the real driver" physical reading :
> the 70 % `storage_read_batch` cost in RR build is **not** driven by per-call `Storage.read` cost growing with offset. It must come from another factor — possibly the number of `Storage.read` calls RR makes (batch iteration pattern), or per-call fixed overhead multiplied by many calls.
> **Consequence for ADR 0001** : the 70 % is potentially actionable within RR. Phase N+1A's scope extends from "fix `list_copy`" to "investigate RR's call pattern to `Storage.read`". ADR 0001 remains viable.

For monolithic this is the applicable reading. The 70 % cost is **actionable within RR** by reducing the number of `Storage.read` calls per build (call pattern change), not by changing what each call does.

### Combined strategic reading

Because production shards in this repo are created by `Storage.append()` (`src/dsm/core/storage.py:80`), which goes through `segment_manager.get_active_segment`, **production shards are segmented, not monolithic**. The Phase 7a.5 fixture used monolithic only because the benchmark harness writes raw JSONL for efficiency. The two verdicts are therefore relevant to different contexts :

- **Phase 7a.5's 70 % figure is not representative of production** — it reflects a monolithic-fixture artefact, amplified by RR's 21-call pagination pattern. Production, which stores segmented shards, would pay the linear-like cost measured in the segmented verdict instead.
- **On segmented shards at 100 k**, the estimated RR `storage_read_batch` cost is ~2.5 s rather than ~10 s. Still significant, but recovers meaningfully without any kernel intervention.
- **On both shard layouts, RR's call pattern matters** : 21 paginated calls vs 1 single call (SessionIndex's pattern) is a 21× multiplier that compounds any per-call cost, and is entirely under RR's control.

Neither case rules ADR 0001 out ; both cases locate the problem in RR's integration with `Storage.read`, not in the kernel's internals under production layout. **No kernel intervention is required by the measurement to unblock gate (iii) investigation.** Whether a follow-up phase then finds that RR can hit gate (iii) under production-segmented layout without kernel changes — that is a separate measurement, not this probe.
